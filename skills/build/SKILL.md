# Build (実装)

## 目的

- テスト駆動開発を標準動作として、小さく安全に実装を進める。
- custom agent chain ではなく、main agent 自身が local に実装を進める入口として使う。

## 使う場面

- 新機能や修正を TDD で進めたいとき
- 実装後にレビューや確認まで一連で進めたいとき
- custom agent orchestration より、main agent 自身で進めた方が軽いとき

## 入力 (任意)

- `design` の出力
- 対象機能の概要
- テスト実行コマンド
- 実装対象ファイル、制約

## 手順

1. 実装タスクを小さな作業単位に分割し、対象と期待動作を明確化する。
2. テストリストを作る。
3. テストリストから 1 件だけ選ぶ。
4. 具体的なテストに翻訳し、失敗を確認する。
5. 最小実装でテストを通す。
6. テストが通った状態でのみリファクタリングする。
7. 気づきをテストリストへ追加し、繰り返す。
8. テスト実行計画と検証への引き継ぎ内容を更新する。
9. 必要なら `reviewer`、`triager`、`validator` を使う。
10. TDD の具体的なリズムが必要なら `references/tdd_twada.md` を参照する。

## 出力フォーマット

- 対象: ...
- 作業単位: ...
- 変更ファイル: ...
- 期待動作: ...
- テストリスト:
  - ...
- 現在のテスト: ...
- 現在のフェーズ: Red | Green | Refactor | Done
- 検証への引き継ぎ: ...
- 次の一手: ...

## 境界

### Always:

- Red → Green → Refactor の順を守る
- テストは 1 件ずつ進める
- 出力は日本語で書く
- 期待動作と検証への引き継ぎを明示する

### Ask first:

- テスト方針や依存関係を大きく変える場合
- 破壊的変更や大量変更が必要な場合

### Never:

- Red の確認なしで実装を始める
- テストを削って無理に通す
