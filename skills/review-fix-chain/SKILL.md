# Review Fix Chain (レビュー修正チェーン)

## 目的

- 人間や他の AI から既に来ているレビュー指摘を custom agent に順番に処理させ、採否判断から修正、確認までを一連で進める。
- レビュー内容の解釈、対応優先度、修正結果、残留リスクを分離して扱う。
- main agent は orchestration と承認境界の管理に徹し、各工程の実作業は custom agent に委譲する。

## 使う場面

- 人間や他の AI から来たレビュー指摘をまとめて処理したいとき
- レビュー指摘の採否と修正を役割分担して進めたいとき
- 修正後の確認まで含めて完走させたいとき
- レビュー本文が既にあり、差分そのものの新規レビューは不要なとき

## 入力 (任意)

- レビュー本文
- 対象差分
- 関連仕様や背景
- 実行可能なテストや lint コマンド

差分だけを渡して新規レビューから始めたい場合は、`code-review` または `reviewer` を使う。

## 手順

1. レビュー対象と入力情報を整理する。
2. `triager` を起動し、各指摘の採用、保留、却下と対応方針を決める。
3. `triager` の出力を引き継いで `builder` を起動し、採用した指摘だけを対象に TDD 前提で修正させる。
4. `builder` の出力を引き継いで `validator` を起動し、確認済み、未確認、残留リスクを整理させる。
5. 修正後に追加レビューが必要な場合だけ `reviewer` を起動し、差分レビューを行わせる。
6. `reviewer` が Must-fix を出した場合は、指摘を `triager` に戻し、再度 `triager` -> `builder` -> `validator` の順に回す。
7. 各段の要点を main agent が必須項目付きの短い引き継ぎ要約に正規化し、次段へ渡す入力を明示してから進む。
8. docs / skills / AGENTS 編集や高リスク変更など承認が必要な作業に入る前は、chain を止めて承認を取る。
9. 最後に、採否結果、実施した修正、残るリスク、次の一手を統合して返す。

## 出力フォーマット

- チェーン対象: ...
- 現在の段階: `triager` | `builder` | `validator` | `reviewer(optional)` | Done
- 直前の受け渡し要約:
  - 直前の段階: ...
  - 対象範囲 / 対象ファイル: ...
  - 実施したコマンド / テストと結果: ...
  - 次段が処理すべき論点: ...
  - 未解決事項 / 前提: ...
  - 承認待ち事項: `あり` | `なし`
- 進行ログ:
  - ...
- 最終結果:
  - triage: ...
  - build: ...
  - validate: ...
  - review: ...
- 未対応の指摘:
  - ...
- 次の一手: ...

## Companion skills (推奨)

- `code-review`
- `validate-fix`

## 境界

### Always:

- custom agent は `triager` -> `builder` -> `validator` を標準順序で起動する
- 追加レビューが必要な場合だけ `reviewer` を起動する
- `triager` を挟まずにレビュー指摘を実装へ流さない
- 前段のアウトプットを必須項目付きで要約して次段へ渡す
- main agent は承認境界と全体進行を管理する

### Ask first:

- AGENTS.md / skills / docs の編集が必要な場合
- 仕様変更の是非判断が必要な場合
- 依存追加や高リスク変更が必要な場合

### Never:

- 採否未整理の指摘をそのまま `builder` へ渡す
- 差分だけを入力にして新規レビュー用途で使う
- 未確認事項を伏せたまま Done にする
- 承認が必要な変更を chain の流れで黙って実行する
