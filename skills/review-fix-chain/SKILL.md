# Review Fix Chain (レビュー修正チェーン)

## 目的

- 人間や他の AI から既に来ているレビュー指摘について、採否判断から修正、確認までを段階的に進める。
- レビュー内容の解釈、対応優先度、修正結果、残留リスクを分離して扱う。
- 承認境界、受け渡し要約、未確認事項を明確にしながら進める。

## 使う場面

- 人間や他の AI から来たレビュー指摘をまとめて処理したいとき
- レビュー指摘の採否と修正を段階分割して進めたいとき
- 修正後の確認まで含めて完走させたいとき
- レビュー本文が既にあり、差分そのものの新規レビューは不要なとき

## 入力 (任意)

- レビュー本文
- 対象差分
- 関連仕様や背景
- 実行可能なテストや lint コマンド

差分だけを渡して新規レビューから始めたい場合は、`code-review` を使う。

## 手順

1. レビュー対象と入力情報を整理する。
2. `triage-review` 段階で、各指摘の採用、保留、却下と対応方針を決める。
3. `build` 段階で、採用した指摘だけを対象に TDD 前提で修正する。
4. `validate-fix` 段階で、確認済み、未確認、残留リスクを整理する。
5. 修正後に追加レビューが必要な場合だけ `code-review` 段階を追加し、差分レビューを行う。
6. `code-review` で Must-fix が出た場合は、指摘を `triage-review` 段階に戻し、再度 `triage-review` -> `build` -> `validate-fix` の順に回す。
7. 各段の要点を必須項目付きの短い引き継ぎ要約に正規化し、次段へ渡す入力を明示してから進む。
8. 並列調査、高リスク判断、品質補強が必要な場合だけ、別 agent / subagent の利用を提案する。
9. docs / skills / AGENTS 編集や高リスク変更など承認が必要な作業に入る前は、chain を止めて承認を取る。
10. 最後に、採否結果、実施した修正、残るリスク、次の一手を統合して返す。

## 出力フォーマット

- チェーン対象: ...
- 現在の段階: `triage-review` | `build` | `validate-fix` | `code-review` | Done
- 条件付き段階: `code-review` (修正後に追加レビューが必要な場合のみ)
- 直前の受け渡し要約:
  - 直前の段階: ...
  - 対象範囲 / 対象ファイル: ...
  - 実施したコマンド / テストと結果: ...
  - 次段が処理すべき論点: ...
  - 未解決事項 / 前提: ...
  - 承認待ち事項: `あり` | `なし`
- 進行ログ:
  - ...
- 最終結果:
  - triage-review: ...
  - build: ...
  - validate-fix: ...
  - code-review: ...
- 未対応の指摘:
  - ...
- 次の一手: ...

## Companion skills (推奨)

- `code-review`
- `validate-fix`

## 境界

### Always:

- 段階は `triage-review` -> `build` -> `validate-fix` を標準順序で進める
- 追加レビューが必要な場合だけ `code-review` 段階を足す
- `triage-review` を挟まずにレビュー指摘を実装へ流さない
- 前段のアウトプットを必須項目付きで要約して次段へ渡す
- 別 agent / subagent は既定で使わず、必要な場合だけ提案する
- 承認境界と全体進行を管理する

### Ask first:

- AGENTS.md / skills / docs の編集が必要な場合
- 仕様変更の是非判断が必要な場合
- 依存追加や高リスク変更が必要な場合

### Never:

- 採否未整理の指摘をそのまま `build` へ渡す
- 差分だけを入力にして新規レビュー用途で使う
- 未確認事項を伏せたまま Done にする
- 承認が必要な変更を chain の流れで黙って実行する
