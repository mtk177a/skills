# Design Changes (変更設計)

## 目的

- 実装前に変更方針を固め、影響範囲と確認方針を明確にする。
- 要件に対して実装可能な設計を作り、単体でレビューや着手判断に使える変更方針と検証戦略を決める。

## 使う場面

- 変更対象や非対象を切り分けたいとき
- 実装前に依存やリスクを整理したいとき
- テストや確認の考え方を先に決めたいとき

## 入力 (任意)

- `scope-request` の整理結果
- 関連ファイルや既存仕様
- 既存コードや構成の理解メモ

## 手順

1. 既存構造とエントリポイントを確認する。
2. 変更対象と変更対象外を分ける。
3. 依存関係、モジュール境界、影響範囲を整理する。
4. リスクと回避策を列挙する。
5. テスト戦略と確認方針を決める。
6. 実装へ進める条件、対象範囲、停止条件を明示する。
7. 変更単位を最小差分に分割し、承認前提で進めるべき点があれば明示する。
8. 高リスクまたは不確実な設計判断が残る場合は、追加確認の必要性を明示する。

## 出力フォーマット

- 設計の要約: ...
- 変更対象: ...
- 変更対象外: ...
- 依存と影響: ...
- モジュール境界: ...
- 主なリスク: ...
- 回避策: ...
- テスト戦略: ...
- 確認方針: ...
- 実装へ進む条件: ...
- 実装の対象範囲: ...
- 停止条件 (`Ask first`): ...
- 変更単位: ...
- セルフレビュー観点: ...

## Companion skills (推奨)

- `plan-safe-change`

## 境界

### Always:

- 変更対象と変更対象外を分ける
- リスクと確認方針をセットで出す
- テスト戦略を明示する
- 実装へ進む条件と停止条件を明示する
- 既存スタイルや設計に合わせる
- 別 agent / subagent は既定で使わず、まずこの Skill 単体で設計判断を返す

### Ask first:

- 依存追加や大きな設計変更が必要な場合
- 大規模整理や破壊的変更が必要な場合

### Never:

- 設計なしで大きな変更へ入る
- 無断で大規模な構造刷新を進める
