# TDD（t-wada）チートシート（要約）

この文書は、TDD 実践用の参照として要点だけを短く整理したもの。

## 基本サイクル: Red → Green → Refactor

- Red: 失敗するテストを 1 件だけ書く
- Red の確認: そのテストが意図した失敗を確実に検出しているかを見る
- Green: 最小限の実装でテストを通す
- Green の優先: 設計の美しさより「通ること」を優先する
- Refactor: テストがグリーンのまま小さく改善する
- Refactor 対象: 実装コードだけでなくテストコードも対象
- この短いサイクルを数分単位で回す

### サイクル詳細

- Red は「これから作る振る舞い」を明文化する段階。失敗を確認できないテストは安全網として機能しないため、必ず失敗を見てから進む。
- Green は「最短距離で通す」段階。仮実装や単純化も許容し、設計の良さは後回しにする。
- Refactor は「動作を保ったまま整える」段階。命名改善、重複除去、関心の分離などを小さく行い、常にテストがグリーンであることを維持する。

## 実践の原則

- テストは常に 1 件ずつ追加する
- Red を必ず確認する
- Green は最小変更で達成する
- Refactor はグリーンの状態で行う
- すべてのテストを毎回実行する
- テストコードもリファクタ対象
- インターフェースは利用者視点で考える
- 「リズム」を守る (小さなステップを素早く回す)
- テストは開発者の不安を減らすために書く
- テストリストを常に更新する
- 外部依存はテストダブルで隔離する
- テストピラミッドを意識する

## よくある誤り

- まとめて複数テストを書く
- Red を経ずに実装を進める
- テストを削除やコメントアウトして通す
- テストの期待値をコピペして通す
- レッド状態でリファクタを混ぜる
- 大きなリファクタを一度に行う

## テストリストの作り方

- 期待する振る舞いを短い箇条書きにする
- 実装設計は後回しでよい
- 途中で思いついたケースは随時追記
- 完了した項目はチェックオフする
