# Code Review (コードレビュー)

## 目的

- 差分をレビューし、正しさ・テスト・安全性・保守性・性能の観点で指摘を整理する。
- 「今すぐ直すべきこと」と「改善提案」を分けて返し、次の対応判断につなげる。
- 対応判断に使えるレビュー結果を整理する。

## 使う場面

- コードレビュー依頼を受けたとき
- 変更をコミットや PR の前に点検したいとき
- 指摘を整理して次の対応へ渡したいとき
- 既存レビュー指摘の処理ではなく、差分そのものの新規レビューをしたいとき

## 入力 (任意)

- 対象差分 (`git diff`、PR diff、コミット差分)
- 関連仕様や背景
- レビューの重点観点

## 手順

1. 変更内容 (diff) を把握する。
2. 正しさ、テスト、安全性、保守性、性能の観点で問題点と改善点を抽出する。
3. 本番事故、セキュリティ、仕様不一致につながる問題を優先して分類する。
4. 指摘ごとに根拠を添え、必要なら再現観点や確認観点を付ける。
5. 出力フォーマットに従って、Must-fix / Should-fix / Nice-to-have に分けて提示する。
6. 高リスク変更やレビュー品質への不確実性が残る場合だけ、追加確認の必要性を示す。
7. 指摘の採否や対応方針が必要な場合は、そのまま判断材料として使える形で未解決事項や確認観点を残す。

## チェック観点

### 1. 正しさ (Correctness)

- 仕様を満たしているか、例外系やエッジケースの扱いは妥当か
- null/空/境界値、順序依存、丸め、タイムゾーン、並行性などの落とし穴がないか

### 2. テスト (Tests)

- 変更点をカバーするテストがあるか
- テストが実装依存になりすぎていないか、モック過多で壊れやすくなっていないか

### 3. セキュリティ (Security)

- 秘密情報の漏洩、インジェクション、権限チェック、入力バリデーションの不備がないか
- ログやエラーメッセージに PII / 機密が出ていないか

### 4. 保守性 (Maintainability)

- 責務分離、命名、重複、複雑さ、例外処理の一貫性に問題がないか
- 既存の設計やスタイルを不必要に崩していないか

### 5. パフォーマンス (Performance)

- 明白なボトルネックや無駄な処理がないか
- 過度な最適化要求は避ける

## 出力フォーマット

- サマリ: ...
- Must-fix:
  - ...
- Should-fix:
  - ...
- Nice-to-have:
  - ...
- テスト手順:
  - ...
- 追加で確認したい点:
  - ...

## Companion skills (推奨)

- `triage-review`
- `validate-fix`

## 境界

### Always:

- 変更内容に基づいてレビューする
- 指摘には具体的な根拠を添える
- 重大度を分けて整理する
- 単体で採否判断に進めるだけの情報を残す
- 別 agent / subagent は既定で使わず、まずこの skill 単体でレビュー結果を返す

### Ask first:

- 対象の変更範囲が不明な場合
- 仕様変更を伴う是非判断まで求められる場合

### Never:

- 指摘の採否を最終決定する
- 実装修正に踏み込む
- 変更内容を見ずに推測で指摘する
