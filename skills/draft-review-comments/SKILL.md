# Draft Review Comments (レビューコメント案作成)

## 目的

- 発見済み・整理済みの指摘を、GitHub PR にそのまま貼れるレビューコメント案へ変換する。
- コメント対象のコード位置、コメントの分け方、review summary の構成を整理する。
- `must` / `should` / `nit` 相当の強さを、丁寧だが実務的なトーンで表現する。

## 使う場面

- `review-changes` や `triage-review-feedback` の結果を GitHub コメントへ落としたいとき
- 自分のレビュー所見を、行コメントと review summary に分けて整えたいとき
- 再レビューで `resolved` / `remaining` / `new` を整理して返したいとき
- stacked PR で、修正タイミング込みの言い回しを整えたいとき
- Approve しつつ、非 blocker のコメントや補足を返したいとき

## 入力 (任意)

- 指摘内容
- 対象コード位置
- 重要度 (`must` / `should` / `nit`)
- 上流 Skill から受けた重要度メモ (`Must-fix` / `Should-fix` / `Nice-to-have`) があればそのまま受け取ってよい
- 文脈
  - 初回レビュー / 再レビュー
  - stacked PR / 通常 PR
  - 後続 PR で修正可か
  - 希望トーン (`柔らかめ` / `標準` / `やや厳しめ`)
- 必要なら前回指摘一覧や今回 diff の要約

## 手順

1. 指摘を 1 コメント 1 論点に分解する。
2. 同じ根本原因から出る複数の症状は、レビュイーの次のアクションが明確になるなら 1 つの論点としてまとめる。
3. 各指摘について、結果の行より直接原因の行を優先して、最も自然なコメント位置を選ぶ。
4. 行コメント、review summary、全体コメント相当のどれで返すべきかを選ぶ。
5. コメントを `事実 → 影響 → 期待` の順で組み立てる。
6. 人ではなくコードを主語にする。
7. `must` / `should` / `nit` の強さを、語尾や依頼表現で調整する。
8. 根拠の強さに応じて、断定、懸念表現、確認質問を使い分ける。
9. 好みレベルの話は blocker 扱いせず、提案として書く。
10. 再レビュー時は `resolved` / `remaining` / `new` を先に整理する。
11. stacked PR では、この PR で直すか後続 PR で許容するかを文面に含める。
12. `must` の指摘は、何が直れば close できるかが分かる粒度で書く。
13. 必要なら review summary や Approve 時の補足コメントを、強さが分かるまとまりで返す。

## コメント種別の使い分け

- 行コメント:
  - 局所実装の問題、直接原因が特定できる論点に使う
- review summary:
  - `must` / `should` / `nice-to-have` の優先順位、全体判断、再レビュー結果の整理に使う
- 全体コメント相当:
  - 後続 PR に送る判断、進め方の補足、Approve しつつ残す非 blocker コメントに使う

## 断定度の目安

- 再現済み、仕様根拠が明確、影響が確定している:
  - 断定してよい
- 静的読解ベースで、影響は強く推測できるが未再現:
  - `〜に見えます` / `〜になりそうです` のような懸念表現を使う
- 仕様前提や意図が未確定:
  - 確認質問に寄せる
- 重大度と断定度は別軸として扱い、`must` でも前提不確実なら断定しすぎない

## 重要度語彙の扱い

- 文面生成では `must` / `should` / `nit` を基本語彙として扱う
- 上流から `Nice-to-have` を受けた場合は、非 blocker の軽微提案として扱い、必要なら `nit` 相当のトーンへ落とす
- review summary では `must` / `should` / `nice-to-have` の整理を残してよい
- 行コメント本文では、ラベルの直書きを必須にせず、語調や依頼表現で強さをにじませる

## 出力フォーマット

- 行コメント案:
  - コメント位置: `path:line`
  - コメント本文: ...
- review summary 案:
  - ...
- 全体コメント相当:
  - ...
- 必要ならトーン違い:
  - 柔らかめ: ...
  - 標準: ...
  - やや厳しめ: ...
- 必要なら Approve 時の補足コメント案:
  - ...
- コメント構成メモ:
  - 分けるべき論点 / まとめてよい論点 / 後続 PR へ送れる論点

## Companion skills (推奨)

- `review-changes`
- `triage-review-feedback`
- `github:gh-address-comments`

## 境界

### Always:

- 1 コメント 1 論点を守る
- 同じ根本原因の論点は、ノイズを増やさない範囲でまとめる
- `事実 → 影響 → 期待` の順で書く
- 人ではなくコードを主語にする
- 好みレベルを blocker 扱いしない
- 断定の強さは根拠の強さに揃える
- 次のアクションが分かる粒度で書く
- 行コメントとして返す場合は、GitHub で行選択して貼れるよう、対象箇所とコメント本文をセットで返す
- review summary / 全体コメント相当では、位置よりも優先順位や進め方が分かる文面を優先する

### Ask first:

- 指摘自体の妥当性が未確定な場合
- コメント対象位置の候補が複数あり、どこに残すかで意味が変わる場合
- review summary だけでよいか、行コメントも必要か不明な場合

### Never:

- 新しい不具合を発見する
- 指摘の採否を決める
- 実装修正を行う
- GitHub API 連携やコメント投稿を行う
