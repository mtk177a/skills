# draft-review-comments 評価メモ

`draft-review-comments` の単体運用を評価するためのメモです。新しい指摘を発見したり採否を決めたりせず、発見済み・整理済みの指摘を GitHub PR 向けコメント案へ落とせるかを見ます。

## 反復 0

- `description` と本文が「レビューコメント文面の提案」に揃っているか
- 新規指摘発見や採否判断に責務が漏れていないか
- 行コメントでは位置とコメント本文がセットで返るか
- 行コメント位置で `path:line` と `path:start-end` を適切に使い分けるか
- 行コメント本文の先頭に正規ラベルが付くか
- `must` / `should` / `suggestion` / `question` / `nit` / `note` の違いが文面に反映されるか
- `Must-fix` / `Should-fix` / `Nice-to-have` を入力ラベルとして受け取り、正規ラベルへ寄せられるか
- 断定の強さが根拠の強さに揃っているか
- 行コメント / レビュー要約 / 全体コメント相当を使い分けられるか
- レビュー要約 / 全体コメント相当で位置を強制しないか
- レビュー要約がラベル別の詳細一覧にならず、短い全体判断に留まるか
- レビュー要約に根拠のある良い点が含まれるか
- まとめるべき関連指摘を過不足なく集約できるか
- `Approve with comments` の文面が実質的な差し止め理由になっていないか
- `Nice-to-have` 相当がマージを止める指摘になっていないか
- トーン未指定時でも既定で圧迫的でない文面になるか
- コメントがレビュイーの能力や姿勢ではなく、コードの挙動と次のアクションに向いているか

## シナリオ

### シナリオ A: 初回レビューのコメント案作成

2-3 件の指摘、対象位置、ラベルを受け取り、強い根拠の `must` と静的読解ベースの `should`、`Nice-to-have` 相当の軽微提案を含めて返す。

評価チェックリスト:

1. [critical] 各行コメント本文の先頭に正規ラベルが付いている
2. 位置と本文が対応している
3. 期待アクションが分かる
4. 静的読解ベースの指摘で断定しすぎない
5. `Nice-to-have` 相当が `suggestion` または `nit` のマージを止めないトーンで返る
6. 各コメントが 1 論点になっている

### シナリオ B: 再レビューの整理

前回指摘と今回差分の要約があり、`Resolved` / `Remaining` / `New` を分けてコメント案とレビュー要約を返す。Approve しつつ補足コメントだけ残す論点も含む。

評価チェックリスト:

1. [critical] `Remaining` を `New` と混同しない
2. `Resolved` を無理に再指摘しない
3. レビュー要約に状態差分が短くまとまっている
4. Approve 時の補足コメントがマージを止める指摘と読めない
5. レビュー要約や補足コメントで位置を無理に要求しない
6. レビュー要約がラベル別の詳細一覧になっていない

### シナリオ C: 前後関係のある PR の後続許容

feature branch / issue branch などから分岐している PR で、この PR 起点だが、後続 PR での修正でもよい論点を含む。全体コメント相当に寄せた方が自然なケースも含む。

評価チェックリスト:

1. [critical] 今回必須か後続許容かが分かる
2. マージを止めない論点を過剰に強くしない
3. 後続 PR へ送れる理由が書かれる
4. 全体コメント相当として返す場合は位置なしでも自然に読める

### シナリオ D: 同じ根本原因の集約

2 つの症状が同じ原因から生じており、別々にコメントするとノイズが増える。

評価チェックリスト:

1. [critical] 根本原因ベースで 1 論点にまとめられる
2. まとめても次のアクションが曖昧にならない
3. コメント位置が結果の行ではなく直接原因の行に寄る

### シナリオ E: 具体条件の明示

条件分岐や null 判定が原因の指摘をコメント化する。`この条件` のような抽象語ではなく、対象の識別子や条件式を文面に含められるかを見る。

評価チェックリスト:

1. [critical] コメントが期待結果、現状挙動、具体条件の順で読める
2. 識別子や条件式が必要な範囲で明示されている
3. 具体化しても文面が冗長になりすぎない

### シナリオ F: 指定 diff に固定したコメント位置

前後関係のある PR や実質 diff 指定があり、現在の作業ツリーや先行 PR / 後続 PR では行番号や原因箇所がずれている。コメント位置、型名、原因説明を指定されたコミット範囲に固定して返せるかを見る。

評価チェックリスト:

1. [critical] `path:line` または `path:start-end` が指定 diff 上の位置に一致する
2. 現在の作業ツリーや先行 PR / 後続 PR の状態を根拠に混ぜない
3. 直接原因の行を優先する
4. コメント本文中の型名や差分説明も指定範囲と整合している

### シナリオ G: 複数行にまたがるコメント位置

複数行の条件式、関数呼び出し、追加 / 削除ブロック全体が原因の指摘をコメント化する。単一行だけに固定せず、GitHub で範囲選択して貼れる位置を返せるかを見る。

評価チェックリスト:

1. [critical] 複数行全体が論点なら `path:start-end` で返る
2. 単一行で十分な論点まで不要に範囲指定しない
3. 範囲指定しても 1 コメント 1 論点が保たれている
4. 指摘の本文が範囲内の式、条件、ブロックと対応している

### シナリオ H: 既定トーンが柔らかめ

トーン指定なしで、`must` / `should` / `suggestion` / `question` / `nit` / `note` が混在する指摘をコメント化する。正しさと期待アクションを保ちつつ、既定で柔らかめの文面になっているかを見る。

評価チェックリスト:

1. [critical] トーン未指定でも詰問調や圧迫的な語尾にならない
2. `must` でも期待アクションが曖昧にならない
3. 作者ではなくコードに向いた文面になっている
4. マージを止めない論点が実質的な差し止め理由に見えない
5. 必要なら短い理由や影響が添えられている
6. 解法を決め打ちしすぎず、レビュイーがよりよい実装を選べる余地がある

### シナリオ I: 質問と参考情報の扱い

前提確認の `question` と対応不要の `note` を含む。どちらもマージを止める指摘として読ませず、確認や参考情報として自然に返せるかを見る。

評価チェックリスト:

1. [critical] `question` が断定的な修正要求になっていない
2. `note` が対応必須に見えない
3. `question` と `note` がレビュー要約でもマージを止めない扱いになっている

### シナリオ J: 簡潔なレビュー要約

複数の行コメント案を返したうえで、レビュー要約は根拠のある良い点、全体判断、主要アクションだけに絞る。詳細なラベル分類や各指摘の再説明を要約側に重複させない。

評価チェックリスト:

1. [critical] レビュー要約が 1-3 文、または最大 3 bullet に収まっている
2. レビュー要約が正規ラベル別の詳細一覧になっていない
3. 根拠のある良い点が短く含まれている
4. 良い点が主要アクションを埋もれさせていない
5. 行コメント側には正規ラベルと具体アクションが残っている
6. マージ可否や残る主要アクションが要約だけで分かる

## 失敗記録の初期項目

- `1 コメントに複数論点が混ざる`
- `ラベルより文面が強すぎる`
- `行コメント本文に正規ラベルが付かない`
- `レビュー要約がラベル別の詳細一覧になる`
- `レビュー要約が指摘だけで心理的に重くなる`
- `根拠のない褒め言葉を書く`
- `根拠より断定が強すぎる`
- `コメント位置と本文が対応していない`
- `同じ根本原因の指摘が細かく分かれすぎる`
- `レビュー要約から次のアクションが落ちる`
- `Approve with comments が実質的な差し止め理由に読める`
- `レビュー要約に path:line を無理に要求する`
- `複数行コメントが単一行に潰れる`
- `単一行で十分なコメントを不要に範囲指定する`
- `Nice-to-have がマージを止める文面へ強化される`
- `前後関係のある PR の後続許容の言い回しが曖昧`
- `指定 diff ではなく現在の作業ツリーにコメント位置を寄せる`
- `既定トーンが要求より強く聞こえる`
- `柔らかめ指定でも詰問調に聞こえる`
- `コメントがコードではなく作者に向く`
- `マージを止めない文面が実質的な差し止め理由に読める`
- `解法を決め打ちしすぎてレビュイーの判断余地を潰す`
