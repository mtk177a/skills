# Diversify Agent Search

## 目的

- agent が現在の案を磨き続けるだけの単一候補改善ループから抜ける。
- agent、workflow、prompt、code の改善を、教訓を残す候補探索、構造的な代替案、保護された評価資産として捉え直す。

## 使うとき

- 同じ方向の試行を 2 回以上繰り返しても意味のある進展がない。
- 改善案が prompt 追記、正規表現、出力正規化、リトライ、局所パッチに偏っている。
- 平均スコアや最新の最良候補だけを見ていて、ケース種別ごとの強みと弱みが不明確。
- 処理順、責務分割、入力分割、リトライ方針、LLM とコードの境界が見直されていない。
- 関連 Skill が、作業が同じ設計近傍へ収束していると検知した。

## 入力 (任意)

- 現在の候補または workflow
- 試行履歴と失敗した分岐
- 評価ケース、スコア要約、テスト結果
- 予算、レイテンシ、安全性、保守性、レビュー容易性などの制約
- 変更してはいけない評価資産

## 手順

1. 現在のアンカーを特定する。どの設計前提、候補、指標の周辺を回っているかを見る。
2. 試行と評価結果から停滞の根拠を列挙する。
3. 保護すべき評価資産を明示する。候補を通すためにケース、期待値、grader、採点ルールを編集しない。
4. 成功した分岐と失敗した分岐の両方を含む候補アーカイブを作る、または更新する。
5. 言い回しの差ではなく、構造差を表す多様性軸を定義する。
6. 構造的に異なる候補を少なくとも 3 つ提案する。
7. 平均スコアだけでなく、ケース種別ごとに候補を比較する。
8. 根拠と多様性に基づいて、次に伸ばす親候補または分岐を選ぶ。
9. 追加実装の前に、予算と停止条件を決める。
10. ローカルに扱える探索なら、この Skill だけで計画を返す。関連 Skill は利用可能で価値がある場合だけ使う。

## 境界

### Always:

- 再利用できる根拠や教訓がある失敗分岐を残す
- 構造的に異なる候補を少なくとも 3 つ提案する
- 平均スコアだけでなく、ケース別の挙動を比較する
- 評価資産と実装候補を分ける
- 他の Skill が未導入でも、この Skill 単体で使えるようにする

### Ask first:

- 評価ケース、期待値、grader、採点ルールの変更が提案される場合
- 探索予算、レイテンシ、実装コストが大きく増える場合
- 候補が依存追加、外部サービス、破壊的変更を必要とする場合

### Never:

- 現在の候補を良く見せるために評価資産を書き換えない
- prompt の言い回し違いだけを十分な多様性として扱わない
- 検証している設計前提を明示せず、現在の候補を磨き続けない
- 基本出力のために別 Skill、subagent、multi-agent workflow を必須にしない

## 補足

- 構造的な多様性には、処理順、責務境界、入力分割、状態表現、リトライ方針、ツール境界、モデル選択、どの部分をコードで固定するかなどが含まれる。
- `break-failure-loop`、`design-changes`、`validate-fix` が利用可能なら companion として使えるが、この Skill は単独で成立する必要がある。
