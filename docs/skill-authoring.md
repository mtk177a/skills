# Skill 作成ガイドライン

## 良い Skill の書き方

良い Skill は、読むエージェントが迷わず使いどころを判断でき、必要な手順や注意点が過不足なくまとまっているものです。

本文は日本語主体で書き、固有名詞・設定キー・既存で定着した技術用語だけを英語で残します。
本文中の一般概念としては `Agent` / `Subagent` ではなく `エージェント` / `サブエージェント` を使います。

特に次を重視します。

- 何のための Skill かが短く明確である
- いつ使うのか、使わないのかが判断できる
- 前提条件、入力、期待する出力が読み取れる
- 特定のエージェントに閉じた記述が必要以上に混ざっていない
- 補助スクリプトや参照資料の責務が明確である

## description の書き方

`description` は説明文ではなく、利用判断の入口です。次のような観点を入れます。

- どんな状況で使う Skill か
- どの作業を助ける Skill か
- 特定のエージェントに依存するなら何に依存するか
- 実行主体や内部オーケストレーションより、利用場面を優先して書く
- 実行主体や役割分担の詳細は、必要なら本文の `目的` や `使う場面` に下げる

避けたい書き方:

- 抽象的すぎて利用場面が見えない説明
- 「便利な Skill」のような主観だけの表現
- Codex 前提なのに、その前提が読み取れない説明
- 利用判断に不要な内部実装の説明を frontmatter に詰め込む書き方

## 共通 Skill とエージェント固有 Skill の分け方

まずはエージェント非依存で表現できるかを考えます。

- 共通 Skill: 複数のエージェントで同じ目的・同じ手順で使えるもの
- エージェント固有 Skill: 利用できるツール、入出力形式、実行手順が特定のエージェントに強く依存するもの

このリポジトリでは分類ディレクトリを増やさず、Skill 名または `description` で差を表現します。
たとえばエージェント固有であることが重要なら、名前や説明にその前提を含めます。

## 外部 Skill を fork / copy する判断基準

外部 Skill は、原則としてこのリポジトリにそのままコピーしません。外部 Skill の導入管理は dotfiles 側の `apm.yml` / `apm.lock.yaml` で行います。

このリポジトリに取り込む判断をしてよいのは、次のような場合です。

- 自分で継続的に改変・保守する必要がある
- 元の Skill をそのまま使うより、再設計したほうがよい
- 自分の運用に合わせて構成や説明を明確に見直したい

取り込む場合も、単純コピーではなく、`frontmatter` と `description` を見直して、このリポジトリの規約に合わせます。

## 秘密情報や非公開情報を入れない方針

次の情報は Skill 本文、補助スクリプト、参照資料、素材のいずれにも含めません。

- API key、token、password
- 顧客名、個人情報
- 社内 URL、社内手順、非公開の運用情報
- ローカル環境にしか存在しない秘密ファイルへの参照

Skill は将来ほかの環境やエージェントからも参照される前提で書きます。
環境依存情報が必要な場合は、変数名やプレースホルダで表現します。

## scripts を含む Skill の注意点

`scripts/` を持つ Skill は便利ですが、移植性を崩しやすいため慎重に扱います。

- シェル、OS、アーキテクチャ依存を最小限にする
- 追加依存がある場合は明記する
- リポジトリ外の絶対パスに依存しない
- 破壊的操作を行うなら、Skill 側でも安全確認を促す
- 生成物やキャッシュをリポジトリに混ぜない

MacBook Pro M1 と Windows WSL の両方で使う前提を崩す script は、必要性が明確でない限り避けます。

## 経験的レビュー資産の置き方

Skill を継続的に改善する場合は、本文だけでなく評価資産も一緒に管理します。
ここでいう経験的レビューは、`empirical-prompt-tuning` Skill に基づく評価運用を指します。

- Skill 単体のシナリオやチェックリストは `skills/<skill-name>/evals/` に置く
- 複数 Skill をまたぐ flow 評価は `docs/empirical-prompt-tuning/` に置く
- 単体改善を優先し、まずは 1 Skill ごとの Iter 0、シナリオ、要件チェックリストを整える
- flow 評価は統合回帰用として扱い、日常改善の主戦場にしない
- まず Iter 0 として `description` と本文の整合、出力フォーマット、単体で完結するかを静的に点検する
- その後、先入観のないサブエージェントで回せるシナリオと要件チェックリストを固定する
- チェックリストには最低 1 つ `[critical]` 項目を置き、成功判定を曖昧にしない
- Skill 本文では、別のエージェント / サブエージェントの利用を既定動作にしない
- 別のエージェント / サブエージェントは、ユーザーが明示した場合や高リスク・高不確実性の補強として提案するに留める

経験的レビューの目的は「作者の主観」ではなく「別のエージェントが迷わず再現できるか」を見ることです。
所感メモよりも、シナリオ、判断基準、失敗パターンの再利用性を優先します。

## Codex 専用 Skill を他のエージェントでも使えるようにする注意

Codex 専用だった Skill を Claude Code や GitHub Copilot でも使いやすくする際は、Codex 固有の表現をそのまま残すかどうかを見直します。

確認ポイント:

- 特定ツール名に依存しすぎていないか
- 依頼の受け方や出力形式が他のエージェントでも通じるか
- 手順が Codex の機能前提になりすぎていないか
- エージェント固有の差分を `description` や本文で十分に説明できているか

Codex 固有の記述を残してよいのは、その前提が Skill の価値そのものになっている場合です。
そうでない場合は、目的と判断基準を中心に書き換え、手段だけを差し替えやすくします。
